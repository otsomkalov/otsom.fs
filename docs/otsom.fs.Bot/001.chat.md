# Chat

Abstract data type to represent a chat, which is used for communication between the Bot and User.

## Options

### Option 1. Record

Base library defines the `Chat` record which will be used by all other Bots

```fsharp
type ChatId =
  | ChatId of int64

type Chat = {
  Id: ChatId
  Banned: bool // Will be used by one of the handlers to check if action from Chat should be processed or not
}

type Update = ...

type UpdateHandler = Chat -> Update -> Task<unit>
```

Pros:

- One type for all the bots
- Minimal boilerplate: no interfaces to implement or generics to carry around
- Easiest for C# interop and JSON serialization
- **Shared Handlers:** Library-provided handlers work out-of-the-box since the `Chat` type is fixed and known.

Cons:

- Addition to satisfy the needs for one bot might be useless for the others
- No way to tailor Chat to Bot needs
- Each extension to the `Chat` will require changes in the library and recompilation with release
- Violates the Open-Closed Principle (OCP)
- Over time, the record can become "God Object" if multiple bots start adding fields to it
- **Shared Handlers:** Every property needed by ANY shared handler MUST be present in the core `Chat` record, even if the bot doesn't use
  that feature.

### Option 2. Interface

Each bot defines its own `Chat`, which should implement the IChat interface.

```fsharp
// Library

type ChatId =
  | ChatId of int64

type IChat =
  abstract Id: ChatId

// Bot

type Chat = {
  Id: ChatId
  Banned: bool // Will be used by one of the handlers to check if action from Chat should be processed or not
}
  interface IChat with
    member this.Id = this.Id
```

Pros:

- Change of `Chat` structure is now purely on the Bot side
- Library functions can work with any implementation using the interface
- Follows standard OO patterns, making it very accessible to C# developers
- **Shared Handlers:** Can be defined against the `IChat` interface.

Cons:

- Each bot needs to define its own `Chat` type
- Requires additional boilerplate code for implementing the interface
- Each handler will have to do type assertion of `IChat` to the Bot specific `Chat` in case it will require any `Chat` specific property
- Lose some F# record-specific features (like structural equality) if using interface-based abstractions instead of the record itself
- **Shared Handlers:** If a handler needs more than just the `Id`, it must either use type assertions (losing type safety) or require a more
  specific interface (leading towards Option 6).

### Option 3. Generic Record

```fsharp
// Library

type ChatId =
  | ChatId of int64

type Chat<'TChatData> = {
  Id: ChatId
  Data: 'TChatData
}

// Bot

type MyBotData = {
  IsBanned: bool
  Language: string
  LastActive: DateTime
}
```

Pros:

- **Type-Safe Extensibility:** Allows bots to define custom data without losing structural equality or record features
- **No Type Assertions:** Handlers are naturally typed to the specific `Chat<'T>`
- **F# Idiomatic:** Feels very natural in F# codebases

Cons:

- Additional generic parameter to all the functions that work with `Chat`
- **Infectious Generics:** Once a function uses `Chat<'T>`, its callers must also be generic or pin down `'T`, which can clutter the API
- Harder to manage in complex dependency injection scenarios compared to interfaces
- **Shared Handlers:** Very difficult to implement generically. A `banHandler` would need to know how to find the `Banned` property inside
  the `'TChatData` generic type, likely requiring complex function pointers or reflection.

### Option 4. Statically Resolved Type Parameters (SRTP)

Use F# inline functions and member constraints to allow "duck typing" at compile time. The library functions accept any type that *looks*
like a Chat (has an `Id` property).

```fsharp
// Library

type ChatId =
  | ChatId of int64

// Inline function that accepts ANY type having an Id member returning ChatId
let inline getChatId (chat: ^T) : ChatId =
    (^T : (member Id : ChatId) chat)

// Bot

type MyBotChat = {
  Id: ChatId       // Fits the constraint
  IsBanned: bool   // Custom field
}
```

Pros:

- **Maximum Flexibility:** No shared hierarchy or interface implementation required
- **Performance:** Calls are resolved at compile time (inlined), avoiding virtual method dispatch overhead
- **Decoupling:** Bot implementations don't depend on a library `IChat` interface, only on the structural shape
- **Shared Handlers:** Can work perfectly! A `banHandler` just needs to be an `inline` function that requires a `Banned` property.

Cons:

- **Complexity:** SRTP syntax is verbose and can be intimidating for developers less familiar with advanced F#
- **C# Interop:** SRTP functions are not easily consumable from C# (they appear as static methods with obscure signatures)
- **Leaky Abstractions:** Error messages for missing members can be cryptic
- **Code Bloat:** Excessive inlining can increase binary size if used heavily across many types
- **Shared Handlers:** Library must expose many `inline` functions, which can't be easily put into a standard collection of handlers unless
  the bot provides a wrapper.

### Option 5. Minimal ID + Service/Context Lookup (Composition)

Define `Chat` strictly as an identifier in the core library. Any additional properties (`Banned`, `Settings`, etc.) are retrieved via
services or providers, not attached to the `Chat` type itself.

```fsharp
// Library

type ChatId =
  | ChatId of int64

// Bot

type IChatSettingsProvider =
  abstract member IsBanned : ChatId -> Task<bool>
  abstract member GetLanguage : ChatId -> Task<string>

// Usage in Handler

let handle (chatId: ChatId) (provider: IChatSettingsProvider) =
  task {
    let! banned = provider.IsBanned chatId
    if not banned then
       // process...
  }
```

Pros:

- **Ultimate Future-Proofing:** The `Chat` type never changes because it is just an Identity
- **Clean Architecture:** Separates "Identity" from "State". State can be stored in Redis, SQL, or memory without changing the core type
- **Lightweight:** Passing `Chat` objects is cheap (just a wrapped `int64`)
- **Scalability:** Better for distributed systems where "State" might be updated by different services concurrently
- **Shared Handlers:** Decoupled. Handlers just need the `ChatId` and a specific service (e.g., `IBanService`).

Cons:

- **Verbosity:** Accessing data always requires a service call or context lookup, rather than just accessing `chat.Banned`
- **Async Complexity:** Properties that were previously simple fields (like `Banned`) might now require async/await calls to fetch
- **Performance Overhead:** Potential for "N+1" problems if many properties are fetched individually via services
- **Shared Handlers:** Bots must provide implementations for every service required by the library's shared handlers.

### Option 6. Flexible Types (#IChat)

Combine interfaces (Option 2) with F# flexible types syntax (`#IChat`). Instead of explicit generic parameters or simple interface types,
the library uses `#IChat` for function parameters, which is a shorthand for `'T when 'T :> IChat`.

```fsharp
// Library

type ChatId =
  | ChatId of int64

type IChat =
  abstract Id: ChatId

// Function using flexible type
let getChatId (chat: #IChat) : ChatId =
    chat.Id

// Bot

type MyChat = {
  Id: ChatId
  Banned: bool
}
  interface IChat with
    member this.Id = this.Id

// Usage in Bot
let myChat = { Id = ChatId 123L; Banned = false }
let id = getChatId myChat // Works without explicit casting
```

Pros:

- **Syntactic Sugar:** More concise than `'T when 'T :> IChat` in function signatures
- **No Explicit Casting:** Unlike using `IChat` directly, you can pass the concrete type `MyChat` to a function expecting `#IChat` without
  an upcast (`:>`)
- **Preserves Concrete Type Information:** Inside the function, it's still treated as the interface, but the caller doesn't have to "forget"
  the original type
- **Shared Handlers (The Mix-in Pattern):** Library can define specific interfaces (e.g., `IBannableChat`) and handlers using
  `#IBannableChat`. Bots only implement the interfaces for the features they want to use.

Cons:

- **Limited Inside the Function:** Inside the function, the value is treated exactly like an `IChat` (or whichever interface it's
  constrained to).
- **C# Interop:** Flexible types are an F# compiler feature. In C#, they appear as the interface type, so C# callers don't get the same "
  no-cast" benefit
- **Inference Issues:** In complex generic scenarios, flexible types can sometimes confuse the F# type inferencer more than explicit generic
  constraints
- **Shared Handlers:** Requires the bot developer to implement multiple small interfaces.

## Long-term Evaluation

| Option                | Maintainability | Extensibility | Performance | Shared Handlers     | Developer Experience    |
|:----------------------|:----------------|:--------------|:------------|:--------------------|:------------------------|
| **1. Record**         | Low             | Low           | High        | Excellent           | High (Initially)        |
| **2. Interface**      | High            | High          | Medium      | Medium              | Medium                  |
| **3. Generic Record** | High            | High          | High        | Poor                | Medium (Generics noise) |
| **4. SRTP**           | Medium          | High          | High        | Good                | Low (Complexity)        |
| **5. Composition**    | High            | Ultimate      | Medium      | Excellent           | Low (Async noise)       |
| **6. Flexible Types** | High            | High          | Medium      | Excellent (Mix-ins) | High (F#) / Low (C#)    |

### 1. Record

In the long term, this will lead to a bloated library that is hard to evolve. Every new bot feature that requires a new field in `Chat` will
force a library release. While great for shared handlers, it forces every bot to carry the data for every handler ever written.

### 2. Interface

A very stable and standard approach. Shared handlers require type assertions unless the interface is already part of `IChat`.

### 3. Generic Record

Excellent for F#-only ecosystems but fails when it comes to shared functionality. It's nearly impossible to write a library handler that
works with any `Chat<'T>` while needing specific fields inside `'T` without reverting to reflection or another abstraction.

### 4. SRTP

A strong contender for shared handlers if performance is critical and everyone uses F#. However, it's hard to organize these handlers into a
reusable collection (like a list of handlers) because they are inherently tied to their specific `inline` implementation.

### 5. Composition

The most robust for large-scale enterprise bots. Shared handlers are easy to write because they depend on services rather than the `Chat`
type itself. However, it's very verbose.

### 6. Flexible Types (The winner for Shared Handlers)

This approach allows the library to provide optional "mix-in" functionality. A bot can be just a `Chat` or a `BannableChat`. The library can
provide `banHandler` which only works on types that implement `IBannableChat`. This is highly modular and preserves type safety without "
Infectious Generics".

## Recommendation

**Suggest Option 6 (Flexible Types with Interface Mix-ins).**

### Why Option 6?

1. **Modular Shared Handlers:** As the library grows, you can add `IBannableChat`, `ILocalizedChat`, `IPermissionChat`, etc. Shared handlers
   are defined against these specific interfaces.
2. **Zero-Overhead for Unused Features:** If a bot doesn't need "ban" functionality, it simply doesn't implement `IBannableChat`. The
   `banHandler` will then be caught at compile-time as incompatible with that Bot's `Chat` type.
3. **Cleanest Handler Composition:** You can mix specific handlers (taking `Chat`) and shared handlers (taking `#IBannableChat`) in the same
   `ButtonClickHandlers` builder seamlessly.
4. **F# First:** It leverages F#'s powerful flexible type system to provide a "duck typing"-like feel with full interface-based safety.
