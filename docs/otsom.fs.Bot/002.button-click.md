# Button click

Abstract data type to represent a button click, which is used for handling user interactions with inline keyboards.

## Options

### Option 1. Record

Base library defines the `Click` record which will be used by all other Bots.

```fsharp
type ButtonClickId =
  | ButtonClickId of string

type Click = {
  Id: ButtonClickId
  MessageId: BotMessageId
  Data: string list
}

type Update = ...

type ClickHandler = Chat -> Click -> Task<unit option>
```

Pros:

- One type for all the bots
- Minimal boilerplate: no interfaces to implement or generics to carry around
- Easiest for C# interop and JSON serialization
- **Shared Handlers:** Library-provided handlers work out-of-the-box since the `Click` type is fixed and known.

Cons:

- Addition to satisfy the needs for one bot might be useless for the others
- No way to tailor Click to Bot needs
- Each extension to the `Click` will require changes in the library and recompilation with release
- Violates the Open-Closed Principle (OCP)
- **Shared Handlers:** Every property needed by ANY shared handler MUST be present in the core `Click` record, even if the bot doesn't use that feature.

### Option 2. Interface

Each bot defines its own `Click`, which should implement the IClick interface.

```fsharp
// Library

type ButtonClickId =
  | ButtonClickId of string

type IClick =
  abstract Id: ButtonClickId
  abstract MessageId: BotMessageId
  abstract Data: string list

// Bot

type Click = {
  Id: ButtonClickId
  MessageId: BotMessageId
  Data: string list
}
  interface IClick with
    member this.Id = this.Id
    member this.MessageId = this.MessageId
    member this.Data = this.Data
```

Pros:

- Change of `Click` structure is now purely on the Bot side
- Library functions can work with any implementation using the interface
- Follows standard OO patterns, making it very accessible to C# developers
- **Shared Handlers:** Can be defined against the `IClick` interface.

Cons:

- Each bot needs to define its own `Click` type
- Requires additional boilerplate code for implementing the interface
- Each handler will have to do type assertion of `IClick` to the Bot specific `Click` in case it will require any `Click` specific property
- Lose some F# record-specific features (like structural equality) if using interface-based abstractions instead of the record itself
- **Shared Handlers:** If a handler needs more than just the basic properties, it must either use type assertions (losing type safety) or require a more specific interface (leading towards Option 6).

### Option 3. Generic Record

```fsharp
// Library

type Click<'TClickData> = {
  Id: ButtonClickId
  MessageId: BotMessageId
  RawData: string list
  Data: 'TClickData
}

// Bot

type MyClickData = {
  Action: string
  EntityId: Guid
  Timestamp: DateTime
}
```

Pros:

- **Type-Safe Extensibility:** Allows bots to define custom data without losing structural equality or record features
- **No Type Assertions:** Handlers are naturally typed to the specific `Click<'T>`
- **F# Idiomatic:** Feels very natural in F# codebases

Cons:

- Additional generic parameter to all the functions that work with `Click`
- **Infectious Generics:** Once a function uses `Click<'T>`, its callers must also be generic or pin down `'T`, which can clutter the API
- Harder to manage in complex dependency injection scenarios compared to interfaces
- **Shared Handlers:** Very difficult to implement generically. A `loggingHandler` would need to know how to find the specific fields inside the `'TClickData` generic type.

### Option 4. Statically Resolved Type Parameters (SRTP)

Use F# inline functions and member constraints to allow "duck typing" at compile time.

```fsharp
// Library

let inline getClickId (click: ^T) : ButtonClickId =
    (^T : (member Id : ButtonClickId) click)

// Bot

type MyBotClick = {
  Id: ButtonClickId
  MessageId: BotMessageId
  Data: string list
}
```

Pros:

- **Maximum Flexibility:** No shared hierarchy or interface implementation required
- **Performance:** Calls are resolved at compile time (inlined), avoiding virtual method dispatch overhead
- **Decoupling:** Bot implementations don't depend on a library `IClick` interface, only on the structural shape

Cons:

- **Complexity:** SRTP syntax is verbose and can be intimidating for developers
- **C# Interop:** SRTP functions are not easily consumable from C#
- **Leaky Abstractions:** Error messages for missing members can be cryptic
- **Code Bloat:** Excessive inlining can increase binary size

### Option 5. Minimal ID + Service/Context Lookup (Composition)

Define `Click` strictly as an identifier in the core library. Any additional properties are retrieved via services or providers.

```fsharp
// Library

type ButtonClickId =
  | ButtonClickId of string

// Bot

type IClickStateProvider =
  abstract member IsProcessed : ButtonClickId -> Task<bool>
```

Pros:

- **Ultimate Future-Proofing:** The `Click` type never changes because it is just an Identity
- **Clean Architecture:** Separates "Identity" from "State".
- **Lightweight:** Passing `Click` objects is cheap

Cons:

- **Verbosity:** Accessing data always requires a service call or context lookup
- **Async Complexity:** Properties might now require async/await calls to fetch
- **Performance Overhead:** Potential for "N+1" problems

### Option 6. Flexible Types (#IClick)

Combine interfaces (Option 2) with F# flexible types syntax (`#IClick`).

```fsharp
// Library

type IClick =
  abstract Id: ButtonClickId
  abstract MessageId: BotMessageId
  abstract Data: string list

// Function using flexible type
let getClickId (click: #IClick) : ButtonClickId =
    click.Id

// Bot

type MyClick = {
  Id: ButtonClickId
  MessageId: BotMessageId
  Data: string list
}
  interface IClick with
    member this.Id = this.Id
    member this.MessageId = this.MessageId
    member this.Data = this.Data
```

Pros:

- **Syntactic Sugar:** More concise than `'T when 'T :> IClick` in function signatures
- **No Explicit Casting:** You can pass the concrete type `MyClick` to a function expecting `#IClick` without an upcast
- **Preserves Concrete Type Information:** Inside the function, it's still treated as the interface, but the caller doesn't have to "forget" the original type
- **Shared Handlers (The Mix-in Pattern):** Library can define specific interfaces (e.g., `ILoggableClick`) and handlers using `#ILoggableClick`.

Cons:

- **Limited Inside the Function:** Treated exactly like an `IClick`.
- **C# Interop:** Flexible types are an F# compiler feature.
- **Inference Issues:** In complex generic scenarios, can sometimes confuse the F# type inferencer.

## Long-term Evaluation

Under the assumption that the `Click` model is stable and won't change over time, the evaluation shifts significantly in favor of simplicity.

| Option                | Maintainability | Extensibility | Performance | Shared Handlers     | Developer Experience    |
|:----------------------|:----------------|:--------------|:------------|:--------------------|:------------------------|
| **1. Record**         | High            | Low           | High        | Excellent           | High                    |
| **2. Interface**      | High            | High          | Medium      | Medium              | Medium                  |
| **3. Generic Record** | High            | High          | High        | Poor                | Medium (Generics noise) |
| **4. SRTP**           | Medium          | High          | High        | Good                | Low (Complexity)        |
| **5. Composition**    | High            | Ultimate      | Medium      | Excellent           | Low (Async noise)       |
| **6. Flexible Types** | High            | High          | Medium      | Excellent (Mix-ins) | High (F#) / Low (C#)    |

### 1. Record (The winner for Stable Models)

When the model is stable, this is the most efficient and developer-friendly approach. It provides a single source of truth, requires zero boilerplate, and makes shared handlers trivial to implement and use. Since success is indicated by the handler's return type (`Option`/`Result`) rather than an internal property, the `Click` record remains focused on the interaction data.

### 2. Interface

A standard approach, but introduces unnecessary abstraction and boilerplate if the model isn't expected to change.

### 3. Generic Record

Adds complexity (infectious generics) for extensibility that isn't needed.

### 4. SRTP

Overly complex for a stable model.

### 5. Composition

The most robust but also the most verbose. Given the model stability, the benefits of decoupling "Identity" from "State" are outweighed by the implementation overhead.

### 6. Flexible Types

Excellent for modular, evolving systems, but introduces more complexity (multiple interfaces) than a simple record when the requirements are fixed.

## Recommendation

**Suggest Option 1 (Record).**

### Why Option 1?

1. **Simplicity:** It is the most straightforward representation in F#.
2. **Data-Centric Logic:** Since handlers rely on the `Data` property and indicate outcome through the return type, a fixed record is all that is required.
3. **Predictability:** Since the model won't change, the risks of "God Object" or violating OCP are minimal.
4. **Optimized Shared Handlers:** Library-provided handlers are easiest to write and consume when they operate on a fixed record type.
5. **Developer Experience:** Zero boilerplate for bot authors and seamless C# interop.
